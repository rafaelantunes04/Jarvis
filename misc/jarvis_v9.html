Por onde começar

Guia de arranque — do teu FastAPI atual ao sistema completo novo

Tens o FastAPI a correr com um modelo fixo. Esta secção responde à pergunta concreta: o que implemento a seguir, e porquê nessa ordem? Cada fase é independente — termina num sistema funcional, mesmo que pares ali.

📍
Onde estás agora
Tens um POST /chat que recebe uma mensagem, chama o Ollama com um modelo fixo, e devolve uma resposta. O modelo não sabe o que disseste antes. Não há modos, não há memória, não há routing. Isso é exatamente o ponto de partida correto — não estás atrasado, estás no sítio certo para começar a construir.
📍
Fase 0 FastAPI + 1 modelo fixo ✓ Feito
O pedido vai do cliente para o FastAPI, o FastAPI chama o Ollama, o Ollama devolve a resposta. Funciona — mas o modelo esquece tudo entre mensagens e comporta-se sempre da mesma forma independentemente do que perguntas.
O que tens
Um endpoint que responde. A fundação está certa.
O que falta
Memória entre mensagens, comportamento adaptável, qualidade de resposta.
💬
Fase 1 Histórico de conversa (SQLite) → Faz isto primeiro ~1 dia
O problema: cada mensagem que mandas é tratada como se fosse a primeira da tua vida. O modelo não sabe que perguntaste outra coisa há 5 segundos atrás.

O que fazes: criar uma tabela conversations no SQLite, guardar cada par pergunta/resposta com um ID de sessão. Antes de chamar o Ollama, carregas as últimas N mensagens da sessão e enviá-as como histórico no payload.

Porquê agora: sem isto, qualquer outra funcionalidade que construas — modos, RAG, tools — fica comprometida porque o modelo não tem contexto de onde a conversa está.
O que desbloqueia
O modelo consegue referir-se ao que disseste antes. "O que disseste sobre X?" passa a funcionar.
Tecnologia
SQLite nativo no Python — sem instalar nada extra. Um ficheiro, uma tabela.
⚡
Fase 2 Streaming de respostas → A seguir ~meio dia
O problema: agora a resposta chega só depois do modelo ter gerado tudo — podes esperar 10-20 segundos a olhar para um ecrã vazio.

O que fazes: em vez de esperares pela resposta completa, o Ollama já suporta streaming nativamente. Mudas o teu endpoint para usar StreamingResponse do FastAPI — os tokens chegam palavra a palavra enquanto o modelo gera.

Porquê agora: muda completamente a perceção de velocidade. Com streaming, parece rápido mesmo quando não é. Implementar mais tarde significa que tudo o que construíres a seguir vai parecer lento até o fazeres.
O que desbloqueia
Experiência de uso comparável ao ChatGPT — respostas progressivas, não espera.
Tecnologia
Já está no Ollama e no FastAPI. Não há nova dependência — é uma mudança no endpoint.
🎛️
Fase 3 Modos e system prompts (YAML/JSON) A seguir ~1 dia
O problema: o modelo comporta-se sempre da mesma forma — o mesmo tom, o mesmo nível de detalhe, o mesmo modelo — independentemente de estares a pedir ajuda com código, a fazer uma pesquisa, ou a querer uma resposta rápida.

O que fazes: criar um ficheiro de configuração por tipo de tarefa (chat, code, research). Cada um tem um system prompt diferente e especifica que modelo usar. O teu FastAPI lê qual o modo ativo e aplica a configuração certa antes de chamar o Ollama.

Porquê agora: é simples de implementar (ler um ficheiro JSON antes de fazer o pedido) mas o impacto na qualidade é imediato e visível. E prepara a fundação para o routing automático que vem depois.
O que desbloqueia
O modelo de código pensa como programador. O modelo de chat é mais conversacional. Usas o modelo certo para cada tarefa.
Tecnologia
YAML ou JSON simples + lógica de leitura no FastAPI. Sem dependências novas.
🖥️
Fase 4 App Desktop (Tauri) — segunda janela de interação A seguir ~1–2 dias
O que fazes: construir a app Tauri com três páginas — homepage, login, e chat. O backend Rust expõe um único comando send_message(username, password, message) via #[tauri::command] que chama o FastAPI e devolve a resposta. O frontend em HTML/CSS/JS consome esse comando com invoke() do bridge Tauri.

Conceito de segunda janela: a app não vive como tab principal — fica posicionada ao lado do que estás a fazer. Chat sempre acessível sem mudar de contexto. Quando fazes uma pergunta enquanto codificas, não precisas de mudar de janela, só de olhar para o lado.

Painéis de visualização (sidebar esquerda): além do chat, quatro painéis consultam o FastAPI para mostrar o estado interno — memória RAG ativa, scores do model strength store por família, log de ferramentas invocadas, e modo e modelo em uso. Tudo só-leitura nesta fase — visualizar, não configurar.

Bot Discord (secundário): implementar a seguir ao Tauri como cliente mobile — recebe mensagens, reactions 👍/👎, comandos /modo. Chama o mesmo FastAPI, sem lógica nova no backend.
O que desbloqueia
Interface de uso real sem abrir o terminal. Ratings disponíveis no desktop e no Discord. Painéis mostram o que o sistema está a fazer internamente.
Tecnologia
Tauri 2 · Rust (reqwest + serde) · HTML/CSS/JS · Bootstrap 3 via CDN · discord.py para o bot.
🧠
Fase 5 RAG — memória semântica das tuas notas Fase seguinte ~3–4 dias
O problema: o modelo só sabe o que foi treinado. Não sabe nada sobre os teus projetos, as tuas decisões técnicas, as tuas notas no Obsidian.

O que fazes: instalar o ChromaDB, correr os teus documentos (notas, código) pelo modelo de embeddings (qwen3-embedding:0.6b via Ollama), guardar os vetores. Antes de cada resposta, o FastAPI procura no ChromaDB os fragmentos mais relevantes para a pergunta e injeta-os no contexto do modelo. O modelo responde com base nas tuas notas, não no ar.

Em duas etapas: primeiro só embedding (recuperação por similaridade). Depois adicionas o reranker (qwen3-reranker:0.6b) que ordena os resultados por relevância real — isto é o estado da arte, não apenas o básico.
O que desbloqueia
"O que decidimos sobre a arquitetura do Jarvis?" tem uma resposta baseada nas tuas notas, não inventada.
Ponto de inflexão
Esta é a funcionalidade com maior impacto percetível. O modelo deixa de parecer genérico.
Tecnologia
ChromaDB (local, sem servidor) + qwen3-embedding + qwen3-reranker via Ollama.
📊
Fase 6 Interaction store + ratings 👍/👎 Fase seguinte ~1 dia
O que fazes: criar uma tabela interactions no SQLite onde cada resposta é guardada com metadados — que modelo foi usado, que modo estava ativo, duração, e (quando avalias) o teu rating 👍/👎. No Discord bot, reagir com 👍 ou 👎 a uma mensagem envia o rating para o FastAPI.

Porquê aqui e não antes: só faz sentido recolher dados depois de o sistema ter qualidade suficiente para usar no dia a dia. Antes das fases 3–5, os ratings refletem problemas de setup, não qualidade real dos modelos.

O que não fazes com estes dados ainda: nada. Só recolhes. O valor vem nas fases 7 e 8.
Ponto de viragem
A partir daqui o sistema começa a aprender. Tudo o que vem a seguir precisa destes dados.
O que desbloqueia
Dados reais sobre o que funciona e o que não funciona no teu uso específico.
🔀
Fase 7 Prompt router — classificação automática de intenção Depois do RAG estar estável ~1–2 dias
O problema: agora tens de mudar de modo manualmente. O sistema não sabe que quando perguntas "porque é que este código dá erro" é uma tarefa de debug, não de chat.

O que fazes: para cada família de tarefas que definiste (code_debug, code_generate, research, quick_chat), escreves 8–12 exemplos de perguntas típicas. Quando chega um pedido novo, o sistema converte-o em embedding e compara com os exemplos de cada família — a família mais próxima é escolhida, e com ela o modelo e system prompt corretos. É automático, sem precisares de dizer /modo.

Biblioteca a considerar: semantic-router (Aurelio AI) — faz exatamente isto e integra com o teu FastAPI sem mudar a arquitetura.
O que desbloqueia
Zero fricção — o sistema escolhe o contexto certo sem intervenção manual.
Estado da arte
Semantic Router, RouteLLM — é exatamente o que os sistemas de produção usam para este problema.
📈
Fase 8 Model strength store — o sistema aprende que modelo usar Depois de 2+ semanas de ratings ~1 dia
O que fazes: criar uma tabela model_strengths onde cada modelo tem um score por família de tarefas. Começas com valores fixos por defeito (bootstrap). Quando chega um 👍/👎, o score do modelo usado nessa interação sobe ou desce para essa família. O mode_router para de ler o modelo do config e passa a perguntar ao strength store: "qual é o melhor modelo para esta família agora?"

O resultado real: ao fim de algumas semanas, o sistema descobriu sozinho que o qwen2.5-coder é melhor para escrever código novo mas o qwen3 é melhor para analisar erros — sem tu configurar nada explicitamente. Apenas os teus ratings a fazer esse trabalho.

Porquê no fim: sem dados de rating (fase 6) e sem routing semântico (fase 7) a funcionar, o strength store não tem informação de qualidade suficiente para tomar boas decisões.
O que desbloqueia
O sistema fica literalmente mais inteligente à medida que o usas — adapta-se ao teu estilo e ao teu stack.
Estado da arte
Versão personalizada do RouteLLM — em vez de usar preferências globais de milhares de pessoas, usa as tuas.
🔬
Fase 9 Optimizer + Judge — sistema auto-melhora prompts e compara modelos Após deadline — sistema estável primeiro ~2–3 dias
Optimizer: quando acumulas 3+ ratings negativos na mesma família numa semana, o optimizer usa essas interações para gerar um system prompt candidato melhor — e testa-o. É a ideia do OPRO (paper Google DeepMind): usar um LLM para otimizar prompts de outro LLM.

Judge: em vez de esperares pelos teus ratings, podes pedir ao sistema para comparar dois modelos numa família com o comando /compare. Um terceiro modelo (idealmente Prometheus-2, treinado especificamente para avaliação) analisa ambas as respostas e escolhe a melhor — o resultado alimenta o strength store.

Porquê no fim: estas ferramentas precisam de um sistema base sólido, com dados reais, para terem valor. Implementar antes é construir sobre areia.
O que desbloqueia
O sistema começa a melhorar os próprios prompts e a comparar modelos sem intervenção humana constante.
Estado da arte
Mixture of Agents (Together AI), LLM-as-Judge (Prometheus-2), OPRO — tudo convergente no que o Jarvis implementa.

A regra de ouro
Cada fase tem de estar a funcionar no uso real antes de avançares para a seguinte. Não implementes a fase 7 se ainda não usas o sistema todos os dias com modos (fase 3). A complexidade tem de ter dados reais para fazer sentido — caso contrário é engenharia no vácuo.

Expansão v3

Três camadas de memória novo

Não é só "ChromaDB para tudo". Cada camada tem um propósito diferente, é injetada em momentos diferentes, e usa tecnologia diferente. A combinação das três é o que torna o Jarvis genuinamente contextual.

Camada 1
Sessão Atual

O quê: o historial da conversa atual — as mensagens trocadas desde que abriste a sessão. É o "contexto imediato" que o modelo usa para manter coerência dentro de uma troca.

Quando é injetada: sempre, em cada pedido. É enviada como o array messages ao Ollama.

Limite: definido pela janela de contexto do modelo (qwen3.5:9b tem 256K tokens — muito mais que o anterior). O FastAPI trunca as mensagens mais antigas quando o limite se aproxima, mantendo sempre as primeiras (instruções) e as últimas (contexto recente).

SQLite
Camada 2
Memória Semântica

O quê: o teu conhecimento acumulado — notas do Obsidian, documentação indexada, código dos repositórios, páginas que mandaste crawlear. É a "biblioteca pessoal" do Jarvis.

Quando é injetada: via RAG, por relevância. O FastAPI converte o teu prompt em embedding, procura os chunks mais próximos no ChromaDB, e injeta os N mais relevantes no context antes de enviar ao modelo. A maioria dos chunks não é injetada — só o que é semanticamente próximo da pergunta atual.

Três sub-índices: general (notas e docs), code (repositórios), web (páginas indexadas via Crawl4AI).

ChromaDB Obsidian
Camada 3
Preferências novo

O quê: factos estáveis sobre ti — linguagens que preferes, estilo de código, ferramentas que usas, convenções dos teus projetos, como gostas de receber respostas. É o que faz o Jarvis não te perguntar as mesmas coisas duas vezes.

Quando é injetada: sempre, como parte do system prompt base. Não é RAG — é sempre incluída integralmente porque é curta e sempre relevante.

Como se atualiza: manualmente via comando (/remember isto) ou o modelo sugere adicionar quando infere algo novo — "notei que usas sempre TypeScript, posso guardar isto?". Guardado em preferences.json lido pelo FastAPI no arranque.

Ferramentas registadas novo

A diferença entre um chatbot e um assistente que faz coisas. Cada ferramenta é uma função Python no FastAPI que o modelo pode "invocar" ao responder — o modelo declara intenção, o FastAPI executa, e devolve o resultado ao modelo para continuar.

📁
read_file / write_file
Lê e escreve ficheiros do sistema de ficheiros local. O modelo pode pedir para ver um ficheiro antes de sugerir edições, e depois escrever diretamente. Caminhos restritos a uma lista de diretórios permitidos.
⚡
run_code
Executa Python, Node.js, ou bash num subprocess com timeout. Devolve stdout, stderr, e exit code. O modelo vê o resultado e pode iterar sem intervenção. É o que permite corrigir código com confirmação real.
🔀
git_status / git_diff / git_log
Chama comandos git e devolve output. Permite ao modelo entender o que mudou desde o último commit, analisar um diff antes de fazer review, ou resumir a história recente de um repo.
🧪
run_tests
Corre pytest (Python) ou jest (Node.js) e devolve os resultados. O modelo pode analisar testes falhados, perceber a causa, e propor correções — completando o ciclo sem saíres do chat.
📝
write_note / update_status
Cria ou atualiza notas no Obsidian. O modelo pode propor adicionar uma decisão ao decisions.md, marcar uma tarefa como feita no status.md, ou guardar um resumo de uma conversa.
🌐
crawl_url / search_web
Crawl4AI para extrair uma página em Markdown. SearXNG para fazer pesquisa privada. O modelo pode pedir para verificar documentação antes de responder, em vez de confiar só no conhecimento de treino.
🔔
send_discord / notify_tauri
Envia mensagem para um canal Discord específico. Usado em automações (notificar resultado de um job) ou quando pedes explicitamente para te lembrar de algo. Ratings 👍/👎 estão disponíveis tanto na app Tauri como no Discord — ambos chamam PUT /interactions/{id}/rating.
🔍
search_memory
Busca semântica no ChromaDB — o modelo pode invocar explicitamente quando sente que existe informação relevante mas não foi injetada automaticamente pelo RAG. Especifica o sub-índice: general, code, ou web.
🌍
http_request
Faz pedidos HTTP a APIs externas com whitelist de domínios. Permite integrar com APIs de trabalho, verificar um endpoint, ou usar webhooks de saída. Mais flexível que ferramentas específicas.
📊
read_logs
Lê os últimos N linhas de um ficheiro de log. Usado em automações de monitorização — o modelo analisa logs de aplicações, scripts ou do próprio Jarvis, e reporta anomalias.
🧩
Como o tool calling funciona na prática — formato correto
O FastAPI envia ao modelo a lista de ferramentas disponíveis (nome, descrição, parâmetros) junto com o prompt. O Ollama usa o formato OpenAI-compatible — importante para a implementação do tool_executor.py:

Resposta do modelo com tool call:
{"tool_calls": [{"function": {"name": "run_tests", "arguments": "{\"pattern\": \"test_auth*\"}"}}]}

Depois de executar, o resultado vai como:
{"role": "tool", "tool_call_id": "...", "content": "3 passed, 1 failed..."}

Este ciclo pode iterar várias vezes antes da resposta final. Nota: arguments é uma string JSON, não um objeto — fazer json.loads(args) antes de usar.
Redesenhado v8

Modos — contexto automático conforme a tarefa redesenhado v8

Cada modo define o que injetar e que ferramentas ativar — não qual modelo usar. O modelo é escolhido separadamente pelo Model Strength Store: um sistema que acumula scores por modelo por família de tarefa, actualizado automaticamente pelos teus ratings e pelas comparações do judge. O modo muda automaticamente quando o router deteta a família da mensagem via embedding similarity — ou podes forçar com /modo código.

O que define um modo (não muda)
  • family — identificador único da família de tarefa
  • system_prompt_version — aponta para o prompt ativo em prompt_versions.json
  • tools_enabled — ferramentas disponíveis para esta família
  • rag_index — qual coleção do ChromaDB usar
  • inject_preferences, inject_project_context — contexto extra
  • embedding_examples — 8-12 frases que classificam esta família
O que o Model Strength Store decide (muda com dados)
  • Qual modelo usar para esta família neste momento
  • Baseado em scores acumulados de 👍/👎 humanos + comparações do judge
  • Um modelo pode ter score alto em code_debug e baixo em research
  • O mesmo modelo pode subir ou descer consoante o teu feedback real
  • Se confiança baixa (poucos dados) → usa modelo bootstrap por defeito
  • Override manual sempre disponível: /modelo qwen3:8b

A separação fundamental: modo ≠ modelo
Na v7 e anteriores, cada modo tinha um modelo fixo no YAML: model: qwen2.5-coder:7b. Era simples mas rígido — não havia forma de o sistema aprender que o qwen3:8b responde melhor a perguntas de debug do que o coder em certas situações, ou que o phi4-mini é suficientemente bom para code reviews simples e 3× mais rápido.

Na v8, o modo define o contexto (prompt, ferramentas, RAG) e o Model Strength Store define o modelo. Os dois sistemas são independentes e evoluem separadamente. Podes mudar o system prompt de code_debug sem tocar na lógica de escolha de modelo — e o modelo pode mudar automaticamente sem tocar no contexto.

Estrutura de configuração de um modo — sem campo model

# data/families/code_debug.json — define contexto, NÃO o modelo
{
  "family": "code_debug",
  "system_prompt_version": "active",        // aponta para prompt_versions.json
  "tools_enabled": ["read_file", "run_code", "git_diff", "run_tests"],
  "rag_index": "code",
  "inject_preferences": true,
  "inject_project_context": true,
  "embedding_examples": [
    "isto dá erro, traceback shows...",
    "porque que não funciona esta função",
    "debug este código, o output está errado"
  ]
  // ← sem "model" aqui. o modelo vem do model_strength_store
}

# O modelo é escolhido assim pelo mode_router (pseudocódigo):
family = prompt_router.classify(message)          // "code_debug"
config = load_family_config(family)               // contexto + ferramentas
model  = model_strength_store.best_for(family)    // "qwen2.5-coder:7b" (por agora)
// quando acumulas dados, pode passar a ser "qwen3:8b" se tiver melhor score

Ideia experimental · v4

Meta-Prompt Engine — Prompts que se Adaptam ao Teu Feedback

A ideia: cada pedido que fazes ao Jarvis não dispara apenas um prompt. Dispara uma cadeia de micro-prompts que avaliam contexto, clareza e intenção antes de chegarem ao modelo principal. No fim do dia, o Jarvis analisa os raciocínios e ajusta os próprios prompts com base no teu feedback — como backpropagation, mas em linguagem natural.

🔬
Estado da arte validado — a tua ideia tem nome
O que describes é Automatic Prompt Optimization (APO) com feedback humano. Existe investigação ativa nisto desde 2023:

TextGrad (Stanford, 2024) — trata feedback do LLM como "gradientes textuais" para otimizar prompts iterativamente, análogo a gradient descent mas em linguagem natural.
OPRO (Google DeepMind, 2023) — usa um LLM como otimizador: constrói um meta-prompt com o histórico de (prompt, score) anteriores e pede ao modelo uma versão melhor.
ProTeGi (2023) — gera gradientes textuais a partir de falhas, aplica beam search para explorar variantes.
DSPy (Stanford, 2024) — compila programas LLM declarativos com teleprompters que otimizam few-shots e instruções automaticamente.

A tua abordagem é mais pragmática e human-in-the-loop: em vez de métricas automáticas, usas o teu próprio julgamento como sinal de treino. Isto alinha-se com RLHF (Reinforcement Learning from Human Feedback) mas aplicado a prompts, não a pesos.

Nó 1 — RAG Check

  • Se sim → passa o contexto diretamente
  • Saída: {"rag_ok": true, "extra_context": "..."}

  • O modelo decide se há ambiguidade que afeta a qualidade da resposta
  • Se sim → pausa e pergunta ao utilizador antes de continuar
  • Se não → continua sem interrupção (caso mais comum)

  • Avalia se o prompt original é vago ou mal formulado
  • Se sim → reescreve ligeiramente sem mudar a intenção
  • A versão reescrita vai ao modelo principal, não o original
  • Saída: {"rewritten": true, "new_prompt": "..."}

  • Output de cada nó com o raciocínio interno
  • Resposta final do modelo principal
  • Timestamp e modo ativo — base para a revisão diária

Revisão diária — a loss function

Fim do dia — análise de 1 a 2 raciocínios
O Jarvis escolhe automaticamente 1 ou 2 interações do dia onde os nós tomaram decisões interessantes (ex: reformulou o prompt, pediu clarificação, fez double RAG). Apresenta-as para revisão.
Passo 1
Seleção automática
Escolhe interações onde algum nó tomou uma decisão não-trivial. Não todas — só as mais informativas.
Passo 2
Apresenta ao utilizador
Mostra o raciocínio de cada nó e pede um sinal simples: 👍 boa decisão / 👎 má decisão / ⊘ ignorar.
Passo 3
Ajuste do prompt
Um LLM analisa o feedback e reescreve o prompt do nó em questão. Guarda ambas as versões com o score associado.
⚙️
Como o ajuste de prompts funciona na prática
Não há pesos numéricos. Cada nó tem um histórico de versões do prompt guardado em prompt_versions.json, organizado por tema (ex: rag_check.code, clarification.projetos). Quando dás 👎 a uma decisão do Nó 2 numa conversa de código, um LLM de revisão analisa: o que correu mal? Reescreve o prompt desse nó só para o contexto "código" com base no teu feedback. Na próxima conversa de código, esse nó usa a versão nova. É OPRO aplicado a nível de nó e tema — a "trajetória" de (prompt_v1, feedback, prompt_v2) é o sinal de treino.

Avaliação honesta da ideia

✅

O que funciona bem

O mecanismo dos sub-prompts é validado pela literatura (TextGrad, ProTeGi). Human-in-the-loop é mais robusto do que métricas automáticas para uso pessoal. A granularidade por tema evita que um ajuste num contexto quebre outro. Implementável sem bibliotecas externas — tudo em Python puro com chamadas ao Ollama.

⚠️

Riscos reais a ter em conta

Os 3 nós adicionam latência (3 chamadas extra por pedido). Se o modelo for lento, isto é perceptível. Solução: usar Phi-4 mini nos nós de meta-análise, reservar o modelo pesado para a resposta final. Além disso, prompts ajustados podem degradar com poucos exemplos — precisas de pelo menos 5-10 feedbacks por tema antes de confiar nos ajustes.

💡

Diferença da abordagem académica

DSPy e TextGrad usam datasets e métricas automáticas. A tua abordagem usa o teu julgamento real em situações reais — o que é mais difícil de escalar mas muito mais alinhado com o teu caso de uso single-user. Não há ground truth externo: és tu o avaliador, o que é simultaneamente a força e a limitação do sistema.

🗂️

Implementar faseado

Fase 1: implementa só o Nó 3 (Prompt Clarity) — o mais simples e com retorno imediato. Fase 2: adiciona o RAG Check. Fase 3: adiciona a revisão diária. Fase 4: fecha o loop com ajuste de prompts. Não implementes tudo de uma vez — cada fase deve ser testada durante pelo menos uma semana antes de avançar.

Prompt Intelligence Engine — prompts que aprendem contigo expandido v6

Expansão do Meta-Prompt Engine da v4. Em vez de um conjunto fixo de prompts, o Jarvis acumula pares prompt-resposta ao longo do dia, classifica cada pedido por família de tarefa, compara respostas entre modelos quando útil, e usa o teu feedback para reescrever os próprios prompts iterativamente. Quatro componentes. A v6 torna explícitas as decisões de implementação: qual abordagem de routing usar, que modelo em cada família, como distinguir as duas tabelas SQLite, e quando cada componente começa a ter valor real.

🔬
Estado da arte — o que existe e o que é sobre-engenharia
DSPy (Stanford/MIPROv2) — o framework académico de referência para optimização automática de prompts. Compila programas declarativos e descobre prompts óptimos via optimizadores. Problema: requer métricas automáticas (ex: % respostas correctas num dataset). O teu sinal é subjectivo e humano — isso não encaixa. Conflitua com o FastAPI custom. Adiciona dependência pesada com magia que dificulta debug. Não usar directamente — roubar as ideias, não o framework.

OPRO (Google DeepMind) — usa um LLM como optimizador: constrói um meta-prompt com o histórico de (prompt, score) e pede ao modelo uma versão melhor. Esta é a abordagem certa para ti — já está em embrião na v4.

LLM-as-judge pairwise — o modelo juiz compara duas respostas e decide qual é melhor. Mais estável do que pontuação individual. Cuidado principal: positional bias (ordem afecta o resultado) — sempre trocar a ordem e só aceitar se ambas concordarem.

Prompt Routing — em vez de um prompt monolítico, separar em famílias especializadas por tipo de tarefa. Classificação por embedding similarity (sem LLM call extra) é o método mais rápido e suficientemente preciso.

O que roubar do DSPy sem usar o framework: (1) guardar pares prompt-resposta com metadata, (2) ter uma função de scoring — o teu 👍/👎 + o judge, (3) um meta-prompt OPRO-style que gera candidatos de prompt melhores com base no histórico. Estes três são suficientes para o teu caso de uso e implementam-se em Python puro sem dependências externas.

Como os scores evoluem — do bootstrap aos dados reais
No início (confiança = "bootstrap"), o sistema usa os defaults da tabela acima. Cada interação com rating 👍/👎 actualiza o score desse modelo para essa família. Ao fim de ~20 rated interactions por família, a confiança passa a "medium" — o sistema começa a desviar-se dos defaults quando os dados sugerem outra escolha. A "high" confiança (30+ rated) significa que os dados dominam completamente. Podes ver os scores actuais com /strengths code_debug.
Como a classificação funciona
  • Ao arranque, o FastAPI embeda os exemplos de cada família com qwen3-embedding:0.6b e guarda em memória
  • Cada mensagem nova é embedada e comparada por cosine similarity contra todos os exemplos
  • A família com maior score médio entre os seus exemplos ganha
  • Se o score máximo for abaixo de 0.45 → fallback para quick_chat
  • Override manual sempre disponível: /modo code força a família
Estrutura do config de família (JSON, sem model)

  • system_prompt_version: "active" — aponta para prompt_versions.json
  • tools_enabled: ["read_file", "run_code", "git_diff"]
  • embedding_examples: ["isto dá erro...", ...] — 8-12 frases
  • rag_index: "code" — colecção ChromaDB
  • ← sem campo model — escolhido pelo model_strength_store
🧭
Abordagem escolhida: embedding similarity (A) — não LLM classifier (B)
Abordagem A — embedding similarity (esta): ao receber uma mensagem, embeddes o texto com qwen3-embedding:0.6b e comparas por cosine similarity contra 8–12 frases representativas de cada família. Sem treino, sem LLM call extra, < 30ms.

Abordagem B — LLM classifier (não usar): um modelo pequeno (ex: phi4-mini) lê a mensagem e decide a categoria. Mais preciso em teoria, mas adiciona 200–500ms por pedido e gasta VRAM.

Optimização de latência: o router corre em paralelo com o início do meta-prompt pipeline — enquanto o Nó 1 (RAG check) já começa, o embedding da classificação corre em asyncio. Latência efectiva: zero percetível.

services/judge.py — comparação pairwise entre modelos
~90 linhas · phi4-mini como juiz · activado com /compare · não por defeito

Quando queres comparar como dois modelos diferentes respondem ao mesmo pedido, o judge corre as duas respostas pelo phi4-mini que decide qual é melhor e porquê. O resultado é guardado no interaction_store, acumulando dados reais sobre qual modelo performa melhor em que família de tarefa.

Input
Mesmo prompt
A tua mensagem vai aos dois modelos em paralelo (asyncio.gather). Ambos processam com o mesmo contexto, RAG, e preferências.
→
Juiz
phi4-mini (2×)
O juiz vê as duas respostas duas vezes com a ordem trocada. Só aceita resultado se ambas as ordens concordarem — elimina positional bias.
→
Output
Vencedor + razão
O modelo vencedor e a justificação do juiz são gravados. Ao longo do tempo, acumulas estatísticas por família: "para code_debug, o coder vence em 71% dos casos."
Prompt do juiz (estrutura)
  • Contexto: família de tarefa, pergunta original
  • Resposta A: [texto do modelo 1]
  • Resposta B: [texto do modelo 2]
  • Instrução: "Qual resposta é melhor para esta tarefa específica? Responde apenas com JSON: {"winner": "A"|"B"|"tie", "reason": "..."}"
  • Corre 2× com A↔B trocado — só aceita se winner concorda
  • Se discordarem → tie — não polui os dados
Quando activar /compare
  • Quando estás a testar um novo modelo (ex: qwen3.5 vs qwen3:8b para projetos)
  • Quando uma resposta te pareceu estranha e queres ver alternativa
  • Periodicamente por família para validar os YAMLs de modo actuais
  • Não activar por defeito — duplica latência e VRAM
  • Acumular 10-20 comparações por família já dá dados significativos
Estatísticas acumuladas — o que o optimizer usa
  • Query SQL simples: SELECT model_used, AVG(judge_score) FROM interactions WHERE task_family = 'code_debug' GROUP BY model_used
  • Ao fim de 30+ comparações por família, tens dados fiáveis sobre qual modelo usar onde
  • O mode_router.py pode automaticamente actualizar qual modelo usa por família com base nestas estatísticas
  • Também acumulas dados sobre latência: às vezes o modelo "pior" é suficientemente bom e 3× mais rápido

services/model_strength_store.py — scores dinâmicos por modelo por família novo v8
~100 linhas · substitui o campo model: nos YAMLs · actualizado por ratings e judge · bootstrap + dados reais

O Model Strength Store é o componente que elimina a rigidez do YAML. Em vez de fixar model: qwen2.5-coder:7b no config de code_debug, o sistema mantém um score por modelo por família — inicializado com valores de bootstrap baseados nas capacidades conhecidas, e actualizado automaticamente com cada rating e comparação. O mode_router consulta sempre o store antes de construir o request ao Ollama.

best_for(family) — algoritmo de seleção
  • Filtra modelos com score acima de threshold por família
  • Se confidence = "bootstrap" → usa bootstrap default sem comparar scores
  • Se confidence ≥ "low" → ordena por score · se empate, usa latência como tiebreaker
  • Sempre verifica VRAM disponível — nunca escolhe modelo que não cabe (via Ollama /api/ps)
  • Se o modelo top não está carregado e outro está em VRAM com score próximo → prefere o carregado (evita swap)
Como o score é actualizado
  • Rating 👍 (+1) → score = α × 1.0 + (1−α) × score_anterior
  • Rating 👎 (−1) → score = α × 0.0 + (1−α) × score_anterior
  • α = 0.15 (suaviza — um único 👎 não derruba o score todo)
  • Judge win → equivale a 👍 com peso 0.5 (sinal mais fraco que rating humano direto)
  • Judge loss → equivale a 👎 com peso 0.5
Bootstrap inicial — o ponto de partida
  • Definido em data/model_bootstrap.json — editável manualmente
  • Exemplo: {"code_debug": {"qwen2.5-coder:7b": 0.8, "qwen3:8b": 0.55, "phi4-mini": 0.3}}
  • Estes valores refletem o que sabes sobre cada modelo antes de ter dados
  • Gradualmente substituídos pelos dados reais à medida que n_rated sobe
  • Podes repor o bootstrap de qualquer modelo com /reset-strength model família
Comandos disponíveis
  • /strengths code_debug — mostra scores e confiança para a família
  • /strengths all — tabela completa de todos os modelos e famílias
  • /modelo qwen3:8b — override manual para a sessão atual
  • /reset-strength model família — volta ao bootstrap default
  • /lock-modelo code_debug qwen2.5-coder:7b — fixa um modelo para a família indefinidamente

⚠️
O que NÃO fazer — armadilhas desta abordagem
Não usar DSPy directamente — foi desenhado para métricas automáticas (ex: % de respostas correctas num dataset). O teu sinal é subjectivo e humano. Conflitua com o teu FastAPI custom. Rouba as ideias, não o framework.

Não optimizar sem rated data — guardar tudo e optimizar com base em volume é ruído. Só interações com 👍/👎 explícito devem alimentar o optimizer e o model_strength_store.

Não fixar modelos por modo no config — é tentador colocar "default_model": "qwen2.5-coder:7b" no JSON de família como forma de simplificar. Faz isso só para o bootstrap; nunca como valor permanente. O model_strength_store deve ter autoridade sobre o config.

Não activar /compare por defeito — duplica latência e VRAM. É uma ferramenta de análise pontual, não um componente sempre activo.

Não fazer mais de 7-8 famílias no início — com poucas interações, ter 15 famílias significa dados insuficientes por família para o strength store ter confiança. Começa com 4-5 e expande quando tens dados para suportar mais.
